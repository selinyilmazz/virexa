# Virexa — Project Architecture

Generated by full codebase inspection on 2026-07-24. This document is a snapshot; keep it updated as the project evolves. It complements (does not replace) `README.md` (setup/ops reference) and `DESIGN.md` (early News Architecture design notes, now partially superseded — see Technical Debt).

---

## 1. Overall Project Structure

Virexa is a production-grade Next.js 15 (App Router) + Supabase application: an AI-enriched developer news platform with a public site and a full internal admin CMS. It follows a strict layered architecture, enforced consistently across ~500 files:

```
Page / Route Handler
  -> Service layer    (src/services/**)      server-only, read services never throw
    -> Repository layer (src/repositories/**) typed Supabase query builders, throw on error
      -> Supabase (Postgres, RLS)
```

Two Supabase clients are deliberately kept separate everywhere in the codebase:

- `src/lib/supabase/server.ts` — request-scoped, cookie-based, respects Row Level Security (RLS). Used for public reads and anything acting as the signed-in user.
- `src/lib/supabase/service-client.ts` — service-role, bypasses RLS. Used only for background persistence (Runtime layer) and admin/privileged write routes. Returns `null` if `SUPABASE_SERVICE_ROLE_KEY` isn't set — every caller treats that as "storage not configured," never as an unhandled error. Never imported from client bundles.
- `src/lib/supabase/client.ts` — browser client, used in `"use client"` auth forms.
- `src/lib/supabase/middleware.ts` — `updateSession()`, the request-scoped client middleware uses to refresh cookies and revalidate the user via `getUser()`.

## 2. Folder Structure

```
src/
  app/                    Routes (App Router): public site + /admin + /api
  components/             React components, one folder per feature domain
  services/               Server-only service layer
  repositories/           Data-access layer (Supabase query builders)
  runtime/                In-process job queue, scheduler, pipeline, health monitor
  lib/                    Shared utilities: supabase clients, ai/, news/, admin/,
                           developer-hub/, explorer/, open-source/, cache/, validation/
  hooks/                  Two shared hooks (useAuth, useBookmarkAction) — most other
                           "hooks" are useSyncExternalStore-based stores in lib/
  data/                   Static/reference content (categories, releases.tsx, countries…)
  types/                  Shared TS types + the Supabase compile-time shim
  i18n/, locales/         next-intl-style i18n system (en/tr/nl)
  styles/                 (mostly superseded by app/globals.css, Tailwind v4 CSS-first)
  middleware.ts           Route protection, maintenance mode, session refresh
supabase/
  migrations/              25 SQL migrations, 0001–0025, additive-only
  diagnostics/             Read-only investigation/runbook SQL scripts
```

No `src/utils/` or `src/constants/` content exists (empty/absent) — utilities live under `src/lib/`. No test files exist anywhere in the repo (no Jest/Vitest/Playwright config) — this is a real gap, not an oversight; see Technical Debt.

## 3. Routing Architecture

App Router, Next.js 15/16 async `params` convention (`params: Promise<{slug}>`). Root layout (`src/app/layout.tsx`) resolves the Supabase session and i18n locale server-side, wraps everything in `I18nProvider` → `AuthProvider`, and renders a global `Footer` (hidden under `/admin` via `ConditionalFooter`). `src/app/admin/layout.tsx` is a fully separate shell for `/admin/*`.

Every public `page.tsx` inspected is a **Server Component**; interactivity is pushed down into imported client components.

Public routes: `/`, `/about` `/privacy` `/terms` `/cookies` (shared `LegalPageLayout`), `/article/[slug]`, `/auth/callback` (route handler), `/bookmarks`, `/categories`, `/category/[slug]`, `/cloud`, `/developer-hub` + 6 catalog sub-routes + `/github` + `/github/[slug]` + `/releases` + `/releases/[slug]`, `/developer-releases` (redirect to `/developer-hub/releases`), `/forgot-password` `/signin` `/signup` `/update-password` (shared `AuthLayout`), `/maintenance`, `/most-read`, `/news`, `/open-source` + `/open-source/topics`, `/resources`, `/reading-history`, `/search`, `/settings`, `/profile`.

A key architectural pattern: **one shared `ExplorerView` template** (`src/components/explorer/ExplorerView.tsx` + `src/components/news-explorer/*`) backs `/news`, `/search`, `/most-read`, `/cloud`, `/resources`, most category pages, and `/developer-hub/releases` — differing only by props (`defaultCategorySlug`, `defaultContentType`, `defaultQuery`, `defaultSort`, `explainMatches`). `/open-source` and `/developer-hub/github` are deliberately **not** built on this template (bespoke pages, different data shape). `/developer-hub/certifications|courses|learning-paths|tools|roadmaps|cheat-sheets` share a second template, `CatalogExplorerView`, over `catalog_items`.

Dynamic routes with `generateStaticParams`/`generateMetadata`/`notFound()`: `article/[slug]`, `category/[slug]`, `developer-hub/github/[slug]`, `developer-hub/releases/[slug]`.

## 4. Database Tables and Relationships

Schema lives in `supabase/migrations/0001`–`0025`, applied in order (manually via SQL Editor per `supabase/diagnostics/verify-repositories-catalog.sql` — there is no automated migration runner). Every migration uses `IF NOT EXISTS`/`OR REPLACE`, safe to re-run.

|---|---|---|---|

| `bookmarks` | Saved items; item_type (article/release/repository/tutorial/resource), item_meta jsonb, denormalized article fields | user_id → `auth.users` | own-row CRUD |
| `user_settings` | 1:1 preferences: theme, reading prefs, notifications/privacy jsonb, timezone | id → `auth.users` | own-row CRUD |
| `article_sources` | Publisher registry: name, domain, trust_score, active | — | public select only |
| `articles` | Core news content; slug/url unique, category, `search_vector` (tsvector, trigger-maintained), subtitle, featured, visible | source_id → `article_sources` | public select only |
| `article_ai` | Versioned AI enrichment: summary/tldr/long_summary/rewritten_article/entities/key_takeaways (jsonb) | article_id → `articles` | public select only |
| `article_metrics` | 1:1 counters: view/bookmark/share/click counts | article_id → `articles` | public select only |
| `admin_audit_log` | Append-only admin action log | actor_id → `auth.users` (set null) | enabled, **no policies** (service-role only) |

| `developer_releases` | Admin-editable release metadata (overlaid onto static content) | — | select where visible |

Notable Postgres functions/triggers: `set_updated_at()` (generic trigger, ~9 tables), `handle_new_user()` (`AFTER INSERT ON auth.users`, auto-provisions `profiles`+`user_settings`, seeds OAuth avatar/name), `articles_set_search_vector()` (`BEFORE INSERT/UPDATE`, replaced a broken generated-column approach after a documented production incident), `search_articles_fts()` (single canonical 12-parameter ranked/filtered/paginated search function — three earlier migrations had accidentally created silent duplicate overloads via `CREATE OR REPLACE` parameter mismatches; `0009` cleaned this up by dynamically dropping every overload first).

## 5. Supabase Authentication Flow

- **Email/password sign-up** (`SignUpForm`): `supabase.auth.signUp()`; if email confirmation is required, redirects to `/signin` with an info toast rather than assuming a session.
- **Email/password sign-in** (`SignInForm`): `supabase.auth.signInWithPassword()`, then redirects to a validated safe path.
- **Google OAuth**: `signInWithOAuthProvider("google", redirectTo)` (`src/lib/supabase/oauth.ts`) → Google → `/auth/callback` route handler → `exchangeCodeForSession(code)` using the request-scoped server client → redirect to `next`. Only Google is enabled; GitHub/Discord/Azure are stubbed for future use.
- **Password reset**: `/forgot-password` → `resetPasswordForEmail()` (always shows success, anti-enumeration) → email link → `/auth/callback?next=/update-password` → `UpdatePasswordForm` confirms a real recovery session via `getSession()`, calls `updateUser({password})`, signs out, redirects to `/signin`.
- **Session management**: cookies are the source of truth, refreshed on every request by `src/middleware.ts` → `updateSession()`. `getUser()` (revalidating against Supabase) is used consistently rather than the cache-trusting `getSession()`, except in the recovery-flow check where `getSession()` is legitimately correct.
- **Open-redirect protection**: every consumer of a `redirect`/`next`/`redirectTo` query param independently validates it starts with `/` and does **not** start with `//` (guards against protocol-relative URL bypass), falling back to `/`.

## 6. Admin Authorization Flow

- `src/lib/admin/is-admin.ts`: `isAdminUser(user)` = `user?.app_metadata?.role === "admin"` — deliberately reads `app_metadata` (service-role-writable only), never `user_metadata` (client-writable, would let a user self-grant admin). The only place `app_metadata.role` is ever set is `src/app/api/admin/users/[id]/route.ts` via `supabase.auth.admin.updateUserById`, itself gated behind an existing admin.
- `src/lib/admin/authorization.ts`: `requireAdminUser()` for Server Components (redirects `/signin?redirect=/admin` if signed out, `/` if signed in but not admin) — called once in `src/app/admin/layout.tsx` so no page repeats the check. `getAdminUserOrNull()` for Route Handlers (returns `null` for either failure case; caller returns 401/403).
- Every `/api/admin/**` route (~25 files) independently re-checks via `getAdminUserOrNull()` — explicit defense-in-depth on top of the middleware-level check, in case a future internal fetch/rewrite bypassed middleware. Every mutating admin route records to `admin_audit_log` after success.

## 7. Middleware Behavior

`src/middleware.ts`, driven by `src/lib/supabase/middleware.ts`'s `updateSession()`:

1. Refreshes the Supabase session/cookies and revalidates the user (`getUser()`).
2. **Admin paths** (`/admin*`): no user → `/signin?redirect=<path>`; signed-in non-admin → `/` (not `/signin`, since re-authenticating changes nothing).
3. **Maintenance mode**: for non-exempt paths (exempt: `/admin`, `/signin`, `/maintenance`, `/api`) and non-admin users, reads `site_settings.maintenance_mode` (fails open to `false` on any DB error) and redirects to `/maintenance` if on.
4. **Protected paths** (`/bookmarks`, `/profile`, `/settings`, `/reading-history`): no user → `/signin?redirect=<path>`.
5. Matcher runs on every route except static assets/`_next`/common image extensions — session cookies refresh site-wide.

## 8. API Routes (`src/app/api/`)

25 route files. Every admin route: `getAdminUserOrNull()` → 403 if not admin → service-role client (503 if unconfigured) → audit log on success.

- **Admin CRUD**: `admin/articles` (+ `[id]`, `[id]/duplicate`, `bulk`), `admin/catalog-items` (+ `[id]`), `admin/collections` (+ `[id]`), `admin/releases` (+ `[id]`), `admin/repositories` (+ `[id]`, `[id]/sync`, `sync-all`), `admin/sources` (+ `[id]`, `bulk`), `admin/settings` (PATCH), `admin/users/[id]` (role/suspend), `admin/users/[id]/reset-password`.
- **Admin read/ops**: `admin/export/[type]` (CSV), `admin/notifications`, `admin/runtime/actions` (safe non-destructive ops: run-pipeline, refresh-cache, recalculate-trending/trust, retry-failed, backfill-images/content/categories/ai-enrichment).
- **Cron**: `cron/news-fetch` (GET/POST, `CRON_SECRET` bearer, runs full ingestion pipeline, `maxDuration=300`), `cron/ai-enrichment` (same auth, fires all 9 AI capability jobs concurrently).
- **Public**: `metrics` (POST, unauthenticated, rate-limited 60/min — bookmark_add/remove, share, reading_time beacons), `reading-history/clear` (POST, session-authenticated — the one route that bypasses the repository layer, calling Supabase directly; flagged as technical debt below).

## 9. Repository Layer (`src/repositories/`)

15 files, one per table/domain, uniform pattern: `create<X>Repository(supabase: SupabaseClient<Database>)` → plain object of typed async methods; every method does `if (error) throw error`. Repositories never construct their own client (take it as a parameter, so the same repository serves both the RLS-respecting and service-role clients) and contain **no business logic** — pure data access. Large `.in()` lookups are chunked (size 100) to avoid PostgREST URL-length errors (a documented production incident). No `GROUP BY`/aggregate support in the query-builder shim, so aggregation happens in application code after a bounded fetch.

Files: `article-repository`, `article-ai-repository`, `article-metrics-repository`, `audit-log-repository`, `bookmark-repository`, `catalog-item-repository`, `collection-repository`, `profile-repository`, `reading-history-repository`, `release-repository`, `repository-repository`, `runtime-job-run-repository`, `settings-repository`, `site-settings-repository`, `source-repository`.

## 10. Services (`src/services/`)

Confirmed: no service file calls Supabase directly — every data access goes through a repository.

- **`services/admin/`** (16 files) — read services never throw (try/catch → `console.error` → safe empty fallback); write-oriented services (`admin-runtime-ops-service.ts`'s recalculate/backfill actions) intentionally throw, since they're invoked from Route Handlers that must report success/failure — documented per-file. `admin-audit-service.ts`'s `recordAuditEvent` is a deliberate exception among "writes": it never throws, so a logging failure can't break the primary action.
- **`services/articles/`** — `article-read-service.ts` (public-site read layer for Home/Category/Search/Article, never throws), `article-metrics-service.ts` (fire-and-forget counters/reading-history writes via service-role client).
- **`services/developer-hub/`** — `developer-hub-service.ts` (unified catalog pool: DB `catalog_items` + live GitHub), `github-explorer-service.ts` (admin-curated `repositories`/`collections`), `release-detail-service.ts` (overlays `developer_releases` onto static content).
- **`services/news/`** — `news-aggregator.ts`, `aggregator-instance.ts`, `live-articles.ts`, `trending-articles.ts` (legacy in-memory layer, superseded for user-facing reads by the DB-backed article-read-service), `providers/*` (RSS/NewsAPI/GNews/HackerNews/Manual, one `NewsProvider` interface).
- **`services/open-source/`** — `open-source-service.ts` (reads `repositories`), `repository-sync-service.ts` (writes live GitHub data into `repositories`, respects `auto_sync`).
- **`services/site-settings/`** — single `getSiteSettings()`, falls back to defaults.
- **`services/ai/`** — see §16 (AI Layer, below, folded into Runtime discussion).

## 11. Shared Hooks (`src/hooks/`)

Only two files: `useAuth.ts` (wraps `AuthContext`, the single source of client-side auth state) and `useBookmarkAction.ts` (shared bookmark-toggle logic used by both article and repository bookmark buttons: signed-out redirect, optimistic toggle, generic error surfacing). Most other "hook-like" logic (`useSettings`, `useProfile`, `useBookmarks`, `useReadingHistory`) lives as `useSyncExternalStore`-based module hooks in `src/lib/*.ts`, not in `src/hooks/` — see State Management.

## 12. Shared UI Components & Design System

**Layout** (`src/components/layout/`): `Header.tsx` (async Server Component, 3-zone grid + search), `CategoryNav.tsx`, `HeaderAuthArea.tsx`/`HeaderBookmarkLink.tsx`/`HeaderNotifications.tsx`/`HeaderThemeToggle.tsx` (all client, `useAuth()`-gated), `Footer.tsx`, `ConditionalFooter.tsx`.

**Providers** (`src/components/providers/`): `AuthProvider.tsx` (the central sync point — see §14), `AuthedThemeScope.tsx` (scopes `.dark` class to a wrapper div on opted-in pages only).

**Shared** (`src/components/shared/`): only `SidebarMiniCard.tsx`.

**Design system**: no shadcn/ui, no Radix, no `class-variance-authority` — confirmed via `package.json` (dependencies are just `@supabase/ssr`, `@supabase/supabase-js`, `next`, `react`, `react-dom`, `zod`). Every component is hand-rolled Tailwind v4 (CSS-first, no `tailwind.config.*`). Conventions: `rounded-2xl`/`rounded-3xl`, `shadow-sm`, slate palette, brand blue `#2f67e8`/hover `#2556c9` (arbitrary-value classes, not a named token). Dark mode: `@custom-variant dark (&:where(.dark, .dark *))` in `globals.css`, applied only via `AuthedThemeScope` on specific pages (Profile/Bookmarks/Settings/Open Source) — not global `<html>` — so most of the site has no dark styling today.

## 13. State Management

Primarily Server Components for data fetching + Route Handlers for privileged mutations, plus one distinctive hand-rolled **client-side per-user cache layer** — not Redux/Zustand/React Query (a code comment in `resource-store.ts` explains React Query wasn't installable in the sandbox environment this was built in).

Core primitive: `src/lib/cache/resource-store.ts`'s `createUserResourceStore<T>()` — TTL cache (60s), in-flight de-dup, `setUser(userId)` reset-on-auth-change, `mutate(updater, persist)` with optimistic update + automatic rollback on failure, exposed via `useSyncExternalStore`. Four stores built on this: `bookmarks.ts`, `profile.ts`, `settings.ts`, `reading-history.ts` (read-only variant). `AuthProvider`'s `syncDataStores()` calls each store's `sync*AuthContext(userId)` whenever `onAuthStateChange` fires — the de facto global-state wiring point. Only one true React Context exists: `AuthContext`.

## 14. Bookmark System

`BookmarkButton` (articles) / `RepoBookmarkButton` (repositories) → `useBookmarkAction` hook → signed-out redirects to `/signin?redirect=<path>`; signed-in calls `toggleBookmark()` (`src/lib/bookmarks.ts`) → optimistic store update → `bookmarkRepository.add/remove` (`bookmarks` table, `upsert`/`delete` on `user_id,item_type,article_slug`) → rollback + generic error message on failure. Article bookmarks also fire a best-effort `/api/metrics` beacon to keep `article_metrics.bookmark_count` in sync. `/bookmarks` page merges Supabase-backed bookmarks with a **separate** localStorage-only store for release bookmarks (`src/lib/release-bookmarks.ts`, since releases aren't `/article/[slug]`-addressable). No dedicated `/api/bookmarks` route — writes go client→Supabase directly under RLS.

## 15. Reading History System

Write path is server-only (no client insert RLS policy exists): `/article/[slug]/page.tsx`, for a signed-in user, fire-and-forget calls `recordArticleRead()` (`article-metrics-service.ts`, service-role client) → `readingHistoryRepository.recordRead()` upserts on `(user_id, article_id)`. Read path mirrors the bookmarks store (`src/lib/reading-history.ts`). Clear action cannot delete directly under RLS, so it calls `POST /api/reading-history/clear` (the one route that bypasses the repository layer — calls `.from("reading_history").delete()` directly rather than through `ReadingHistoryRepository`, which has no delete method).

## 16. Settings System

Table `user_settings` (one row/user). Store `src/lib/settings.ts` (same resource-store pattern). Fields: General (language, timezone, preferredCategories), Reading (summaryLength, openLinksInNewTab, readingProgressBar/rememberScrollPosition/readingWidth — mostly saved-only, not yet applied to UI), Appearance (theme — the one field with a real visible effect), Notifications (9 flags), emailPreferences, Privacy (profileVisibility, analyticsConsent, personalizedRecommendations, trackSearchHistory, trackReadingHistory). UI: `/settings` → `SettingsForm` with 6-category sidebar nav (`?category=`). Privacy panel includes Clear Reading History / Clear Bookmarks / Export Data (client-side JSON download) and a Danger Zone (account deletion currently only clears local caches + signs out — full `auth.users` deletion explicitly deferred per code comment). Language change also sets `virexa_locale` cookie and triggers `router.refresh()`.

## 17. Profile System

Table `profiles` (one row/user, provisioned by the `handle_new_user()` trigger). Store `src/lib/profile.ts` — `email`/`joinDate` come from the Supabase auth user object (not a `profiles` column), injected via `syncProfileAuthContext`. `/profile` shows `ProfileHeader` + `ProfileStatsSection` (bookmarks/reading count/releases-viewed/top-category, computed client-side) + `ProfileOverviewTab`. Editing lives exclusively in Settings → Account (`ProfileEditForm`, Zod-validated, optimistic save/rollback) — one single editing surface reused between `/profile`'s "Edit Profile" link and `/settings?category=account`.

## 18. Developer Hub Architecture

Three data sources feed one unified `CatalogItem` shape via `developer-hub-service.ts`'s `buildCatalogPool()`:
1. **`catalog_items` table** — certifications/courses/learning-paths/tools/roadmaps/cheat-sheets, full admin CRUD at `/admin/catalog-items`. This **replaced** the formerly-hardcoded `src/data/developer-hub.ts`, which no longer exists — a completed migration.
2. **Live GitHub API** (`src/lib/developer-hub/github.ts`, `getTrendingGithubRepos()`) — a fixed 12-repo tracked list, 15-min TTL cache, fail-open. Used only for the Dev Hub landing page's trending strip, not the dedicated GitHub Explorer.
3. **Real `articles` rows** (`contentType: "release"`) for `/developer-hub/releases`, deliberately excluded from the catalog pool.

Sub-pages: landing dashboard, 6 catalog sub-routes (all thin `CatalogExplorerView` wrappers), GitHub Explorer (its own template, see §19), Releases (reuses `ExplorerView`). `src/lib/developer-hub/continue-learning.ts` tracks clicked resources in localStorage only (explicitly not a fabricated completion percentage).

## 19. GitHub Explorer Architecture

Redesigned from a GitHub-trending clone into a fully DB-backed "Developer Knowledge Library." `repositories` table (PK `"owner/repo"`) built across 4 migrations (0018 base, 0023 extended stats, 0024 editorial fields + `collections`/`collection_repositories`, 0025 media). Data layer: `github-explorer-service.ts` (filter/sort/paginate, featured collections, hero carousel, related/alternative repos, all fail-open with a `dataUnavailable` flag distinguishing real errors from healthy-empty results). Detail page (`/developer-hub/github/[slug]`) combines the DB row with **live**, uncached GitHub fetches for README excerpt and recent releases. Collections are admin-curated many-to-many groupings via `collection-repository.ts`, distinct from the fixed `category` taxonomy. Sync: `repository-sync-service.ts` pulls live GitHub stats into `repositories`, respecting each row's `auto_sync` flag.

## 20. Developer Releases Architecture

Two things share the name: (1) `/developer-hub/releases` list = real `articles` rows via the shared `ExplorerView`; `/developer-releases` is now a pure redirect to it. (2) `developer_releases` table = admin-editable metadata for per-technology detail pages at `/developer-hub/releases/[slug]`, **overlaid onto** rich hand-authored static content in `src/data/releases.tsx` (748 lines, intentionally left static, not turned into a CRUD form) via `release-detail-service.ts`'s `getReleaseDetail()`, using the service-role client so a `visible=false` row can be distinguished from "no row." Admin CRUD for the table at `/admin/releases` is fully wired.

## 23. Important Coding Conventions

- Repository = data access only, throws on Supabase error, takes `SupabaseClient` as a parameter (never constructs its own).
- Read services never throw (catch → log → safe fallback); write services that must report success/failure to the UI intentionally throw, and this is documented per-file where it deviates from the norm.
- Two Supabase clients, strict separation; service-role key never reaches client bundles.
- `app_metadata.role`, never `user_metadata`, for any authorization decision.
- Every `redirect`/`next` query param independently validated against open-redirect (`/` prefix, not `//`).
- Optimistic client mutation + automatic rollback is the standard pattern for all four per-user resource stores.
- Large `.in()` queries chunked at 100 to avoid PostgREST URL-length failures.
- `CREATE OR REPLACE FUNCTION` migrations must match parameter lists exactly or they silently create a new overload (a real incident, fixed in migration 0009) — worth remembering for any future `search_articles_fts`-style function change.
- Mock/placeholder data is rare and, where it exists, is explicitly self-documented in a code comment as intentional (see §26) — never silently mixed with real data.

## 24. Files That Should Never Be Duplicated

- `src/types/supabase-shims.d.ts` — temporary compile-time Supabase type shim; deleting it introduces 100+ TypeScript errors until real network-installed types are available. Only extend, never remove, per project rules.
- `src/lib/supabase/server.ts`, `service-client.ts`, `client.ts`, `middleware.ts` — the four canonical Supabase client constructors. Any new server code needing Supabase access should import one of these, never instantiate a new client.
- `src/lib/cache/resource-store.ts` — the one generic per-user cache engine; a new per-user cached resource should be built on this, not a bespoke `useState`/`useEffect` fetcher.
- `src/lib/admin/is-admin.ts` / `authorization.ts` — the one authorization check; never re-implement role checking inline in a page or route.
- `src/components/explorer/ExplorerView.tsx` and `src/components/developer-hub/CatalogExplorerView.tsx` — the two shared list/filter/paginate templates; a new news-like or catalog-like listing page should extend these via props, not fork them.
- `src/hooks/useBookmarkAction.ts` — the one bookmark-toggle hook; both article and repository bookmark buttons already share it.
- `src/repositories/*.ts` — one file per table; a second repository for the same table would violate the architecture directly.
- `middleware.ts` — one root middleware; route protection additions belong inside its existing path-list constants, not a second middleware file.

## 25. How Major Systems Interact

- **Auth → everything**: `AuthProvider` (client Context) is the hub. On every Supabase `onAuthStateChange` event it calls `syncDataStores()`, which re-points all four resource stores (bookmarks/profile/settings/reading-history) at the new user. `middleware.ts` independently refreshes the session/cookies on every request and gates `/admin` and the four protected paths server-side, so client and server auth state are kept in sync through the same underlying Supabase session cookie.
- **Public pages → services → repositories → Supabase**: Server Components call service functions directly (no client-side fetch), which is why reads never throw — a failure degrades the page instead of crashing it (empty states, not error boundaries, for data problems).
- **Article detail page ties together four systems at once**: on load it calls `article-read-service` (display), fires `incrementArticleView` and (if signed in) `recordArticleRead` (metrics + reading history, both service-role, both fire-and-forget), and its `ShareButtons`/`BookmarkButton` wire into the bookmark system — all four systems converge on one page without any of them depending on the others directly.
- **Runtime layer → AI layer → article_ai / articles**: the cron routes (`/api/cron/news-fetch`, `/api/cron/ai-enrichment`) are the only production trigger of the Runtime engine (the in-process scheduler is a self-hosted-only opt-in that nothing calls automatically). `news-fetch` is the only job of the ~19 registered types that persists new articles; the 9 AI-capability jobs each independently call `ai-enrichment-runner.ts`, which itself wraps the provider-agnostic `AIService`, sharing one implementation with the Admin Runtime panel's manual "Backfill AI Enrichment" button — no duplicated enrichment logic between the automatic and manual paths.
- **Admin writes → audit log → Admin Dashboard/Activity feed**: every admin mutation route calls `recordAuditEvent()` after success; the Dashboard's Recent Activity feed and `/admin/audit-log`-equivalent (folded into Dashboard) both read `admin_audit_log` through the same repository, so there's a single source of truth for "what changed and who did it."
- **GitHub Explorer, Open Source Explorer, and the Developer Hub catalog strip all read the same `repositories` table** but through different service functions with different filter defaults (`github-explorer-service.ts` for the curated Explorer, `open-source-service.ts` for the bespoke Open Source page, `developer-hub-service.ts`'s live-GitHub path for the landing-page trending strip) — a change to the `repositories` schema affects all three and should be checked against all three service files.
- **i18n threads through both server and client**: `resolve-locale.server.ts` (cookie or signed-in user's `user_settings.language`, cache()-wrapped) determines the locale for every Server Component render; the Settings page's language selector writes both the cookie and, if signed in, `user_settings` via the settings resource-store, then triggers `router.refresh()` to re-render server content in the new locale.

## 26. Current Technical Debt

- **No automated tests exist anywhere in the repo** (no Jest/Vitest/Playwright, no `*.test.*` files) — the project relies entirely on `npx tsc --noEmit`, `npm run lint`, and manual verification. This is the single biggest gap for a project this size.
- **`src/types/supabase-shims.d.ts` is a hand-maintained stand-in for real Supabase types**, needed because the original sandbox this project was built in had no network access for `npm install`. It should be deleted (per its own header comment) the moment real, published `@supabase/supabase-js` types are confirmed compatible — but per project rules, do not delete it preemptively.
- **`README.md`'s Runtime section understates the job registry**: it says "13 job types" / "14 with hn-sync," but `src/runtime/types.ts` and `jobs/index.ts` currently define and register **19** job types (9 of which are the individually-scheduled AI enrichment capabilities). The README should be updated to avoid confusing a future contributor about the pipeline's real scope.
- **`README.md`'s Database section only documents migrations 0001–0003**; 22 more migrations (repositories, developer releases, site settings, catalog items, collections, full-text search fixes) exist and aren't reflected in that section. This document (§4) is the more current source for schema.
- **One repository-layer bypass**: `src/app/api/reading-history/clear/route.ts` calls Supabase directly instead of through `ReadingHistoryRepository` (which has no delete method). Low risk (isolated, well-understood), but inconsistent with the stated architecture rule.
- **`noUncheckedIndexedAccess` is disabled in `tsconfig.json`**, surfacing ~24 known, mostly low-risk array-access call sites across the codebase that assume indexed access always succeeds. Left disabled deliberately (needs case-by-case review), not blindly patched.
- **No Content-Security-Policy header** — deliberately deferred (see `next.config.ts`'s comment) until one can be authored and tested against the real deployed build's script/style fingerprints.
- **Rate limiting is per-process, in-memory** (`src/lib/rate-limit.ts`) — provides no hard cap on a multi-instance/serverless deployment; would need a shared store (Redis/Upstash) for a strict guarantee.
- **Two narrow, explicitly self-documented mock-data sidebar widgets remain**: `src/lib/explorer/developer-pulse-data.ts` (Explorer "Developer Pulse" topic stats) and `src/lib/open-source/pulse-data.ts` (Open Source "today" delta metrics). Both are clearly commented as intentional placeholders pending a real data source — not accidental leftover mock data, and not mixed into any primary content.
- **`DESIGN.md`'s "Known gaps" section is now stale** — it describes a phase where Home/Article/Profile/Settings/Bookmarks "still read only from `src/data/*` mock data," which is no longer true (all of these are Supabase-backed today per this inspection). Worth a note or removal so it doesn't mislead a future reader.
- **Admin Users Management fetches up to 1,000 users into memory** for search/sort/filter (`admin-user-service.ts`); fine at current scale, would need pagination-at-the-API-level at much larger user counts.
- **Analytics time-series data is explicitly best-effort** where the schema doesn't retain daily history — documented per-metric in `admin-analytics-service.ts`'s doc comments, not a hidden limitation.

## 27. Potential Improvements

- Add an automated test layer (even a minimal Vitest + Testing Library setup for the repository/service layer's pure logic, and Playwright smoke tests for the auth flow and one admin CRUD flow) — the layered architecture (repositories/services as plain functions taking a client) is already well-suited to unit testing without a running Supabase instance (mock the `SupabaseClient` parameter).
- Update `README.md`'s Runtime and Database sections to match current reality (19 job types, all 25 migrations) — low effort, high value for onboarding.
- Consider consolidating or removing the individually-scheduled legacy jobs (`rss-sync`/`newsapi-sync`/`gnews-sync`/`hn-sync`) that only touch the superseded in-memory `live-articles` cache and no longer feed any database-backed page — `schedule-definitions.ts` already documents this as a known cleanup candidate.
- Fix the `reading-history/clear` route to go through a new `ReadingHistoryRepository.clear()` method for consistency with the bookmark system's equivalent `clear()` method.
- Real daily-delta ingestion (or an honest "N/A" state) to retire the two placeholder pulse-data widgets once there's a real source for them.
- Extend dark mode beyond `AuthedThemeScope`'s current opt-in pages if a full-site dark theme is ever desired — the Tailwind v4 class-based variant is already wired, it's only application that's partial.
